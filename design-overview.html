1What we're building

A teammate you @mention in Slack, that does real work and remembers your team.

Anthropic just launched Claude Tag: you type @Claude in a Slack channel and hand it a task. It works in the thread where everyone can watch, remembers the channel over time, can jump in on its own, and can keep working on something for hours. It's like hiring a coworker who lives in Slack.

The catch: Claude Tag only runs on Anthropic's servers, only uses Anthropic's models, and reads all your Slack messages on someone else's computer. We're building the open version of the same idea — one you install and run yourself.

The one-sentence version A Slack bot that, when tagged, runs an AI "agent" with access to your tools, keeps a memory for each channel, and can act on its own — all on infrastructure you control, with whatever AI model you choose.

Think of it like the difference between renting a coworker from an agency (Claude Tag) and hiring your own that sits at a desk in your own office (this project). Same job — but you hold the keys.

2Why ours is different (the "wedge")

Three things the official Claude Tag can't do — and our whole reason to exist.

🏠

Self-hosted

Runs on your own server or laptop. Your Slack messages and data never leave machines you control. Answers the fear: "I don't want a third party reading everything."

🔌

Any model

Plug in Claude, GPT, Gemini, Llama, or a local model — and switch anytime. You're never locked to one AI vendor or their pricing.

🔍

Open & auditable

Open source. You can see exactly what it does, scope what each channel can touch, and keep a full log of who asked for what.

Where this came from We're growing it out of an existing bot ("Henry") that already lives in one Slack channel and investigates customer-support bugs. That bot is already ~60% of the machinery — we're generalizing it. We also studied getsentry/junior (an open-source, plugin-based Claude Tag by Sentry) for its architecture, and got a second opinion from a different AI (Codex) on the big calls.

3The big picture: what happens when you tag it

Follow one Slack message through the system. Each box is a stage; the data flows top to bottom.

Slack
The front doorA message arrives. We use Slack's official toolkit (Bolt). The bot sees every message in channels it's in.
↓
Gate
"Should I act?"A cheap, fast check. Tagged directly → yes. Otherwise a small model decides if it's worth jumping in (this powers the proactive mode). Most messages are ignored — on purpose.
↓
Orchestrator
Set up the workspaceLocks the channel so two requests don't collide, loads that channel's identity (its instructions + which tools it may use), and loads its memory.
↓
Agent
The thinking loopThe AI plans, calls tools, reads results, repeats — until the task is done. Built on Pydantic AI; the model is supplied through LiteLLM so it can be any LLM.
↓
Integrations
The handsThe tools the agent can use — GitHub, your ticket system, logs, etc. Credentials are attached here by the host, never shown to the AI.
↑↓
Slack
Back to the threadThe answer (or progress updates) post into the same Slack thread, visible to the whole team.

Two helpers sit alongside this pipeline and are part of the product, not add-ons:

Memory
Per-channel memoryA small, growing notebook for each channel (facts learned, work done, open tasks), stored in a database.
Scheduler
Durable tasksA to-do list of future/long work so the agent can keep going over hours or days and survive restarts.
Wrapping everything: Governance Spending limits per channel/company, an audit log of every action, and rules for which channel can use which tools. This is what makes it safe to give a whole company.

4Core vs. integrations

The most important architecture decision: what's built-in forever vs. what plugs in.

We split the system into a core that we own and ship, and an integration layer for connecting tools. The agent's abilities (memory, scheduling, jumping in) are core — they ARE the product. The tools (GitHub, Linear…) are pluggable.

🛡️ Governancespend caps · audit log · per-channel tool permissions
🧠 Core capabilitiesSlack identity & routing · per-channel memory · scheduler/durable tasks · proactive "gate" · permissions
⚙️ Agent runtimePydantic AI loop · LiteLLM (any model) · credential store · private OAuth
🔌 Integration layera simple internal "Integration" contract — tools plug in behind it
🧰 ToolsGitHub · ticket system · logs · MCP servers · your own Python functions
Why not just copy Sentry's "plugins"? Sentry's junior makes everything a formal plugin (even memory). That's great once you have many integrations and outside contributors. We're not there yet, so for now we use a lightweight internal Integration contract — the same "seam," without the heavy public plugin format. We graduate to full plugins later, when we actually need them. (This was the key takeaway from the Codex second opinion.)

5Integrations & secrets — the part that matters most

How the bot uses your tools without ever leaking your passwords.

What an "integration" is

An integration bundles three things: (1) what it can do (the tools, e.g. "search GitHub code"), (2) how it logs in (the credentials/OAuth), and (3) a short instruction snippet telling the AI how to use it well. A strict rule keeps these clean: the instructions never handle passwords, and the passwords never reach the AI.

The golden security rule

Secrets are never shown to the AI model. The AI says "call the GitHub search tool with this query." The host program — your code — is the one that actually attaches the GitHub token and makes the call. The token is never put in the prompt, the tool's inputs, the results it sees, or the logs. The AI gets the answer, never the key.

Codex's sharp warning, which we're taking to heart: there's a difference between "the secret isn't sent to the AI" (what we do — and it's enough for privacy) and "the secret is impossible for any tool code to read" (a much heavier setup with sandboxes that we'll only add later, for untrusted plugins). We're careful not to over-promise the second.

Connecting your account: private, pause-and-resume

The slickest idea we're borrowing from Sentry's junior. Say you ask the bot to do something in Linear but you've never connected Linear:

1
You: "@Claude, file this as a Linear ticket."
↓
2
Bot notices there's no Linear connection. It pauses the task and direct-messages you a private login link (never posted in the public channel).
↓
3
You click it, approve access. The token is stored safely on the host, tied to your identity.
↓
4
The bot resumes the paused task automatically and posts the finished ticket in the thread.

Why not Composio, and why not real CLIs

No Composio (for the core)

Composio is a hosted service that holds your logins and runs your tool calls on their cloud. That breaks our "self-hosted, private" promise (and they had a breach). We use the open standard (MCP) + our own tools instead, and may offer Composio as an optional add-on later.

No real command-line tools

Junior runs real CLIs (like gh) in a sandbox. Powerful, but heavy and fiddly. For a chat teammate, plain API/function tools are simpler, safer, and fit our Python stack — so that's what we use.

6Running code safely (the sandbox)

Two kinds of "doing" — and only one needs a locked room.

The bot does real work, not just lookups. But "doing" splits in two, with very different safety needs:

🎫 Taking actions in other apps

Open a GitHub PR, create a Linear ticket, post a message. The bot fills in a form and our code presses the app's official button. It isn't running its own program — so the safety control is permissions (which buttons it may press), not a sandbox.

🧪 Running its own code

Write a script and run it, run the test suite, crunch a data file. Now the bot is driving a computer and could do anything — so this happens inside a sandbox: a sealed, throwaway computer that can't touch your real files or other teams' data.

The "create a PR" example, both ways Quick: the bot writes the change as text and our code pushes it through GitHub's API — no sandbox needed. Good: the bot checks out the repo, edits files, and runs the tests to confirm the fix works before opening the PR — that needs the sandbox. Once we want the bot to verify its own work, the room is in.

The room is swappable — pick your trade-off

🪶 Lightweight built-in (default)runs on a laptop, no extra setup · model-agnostic · our pick for "just run it"
🏢 Managed: E2B · Daytona · Modalstrongest isolation, scales itself, GPU — flip on with one config change
Why a built-in default — and not force E2B / Daytona / Modal Those are excellent strong rooms and fully supported — but they're mostly cloud services. Forcing one would quietly break our "runs on your own box, data stays yours" promise. So the default is a tiny built-in room that needs nothing; teams who want managed power switch with one line. Same pluggable idea as the AI model and the tools.

7How memory works

Each channel gets its own memory — and that's what makes handoffs and "it remembers us" possible.

For every channel we keep two simple things in a database:

📒 The memory notebook

A growing document of what the bot has learned in this channel — durable facts, a running summary of work done, and open tasks. The AI reads and updates it as it works.

⚡ The quick snapshot

A tidy summary we paste into the AI's instructions at the start of every turn, so it has context immediately — before it even opens the notebook.

Crucially, this is done in a model-agnostic way: instead of Anthropic's special memory feature (which only works with Claude), we give the AI plain "read memory / write memory" tools that any model can use. We start simple (no fancy search database) because a single channel's notes easily fit in the AI's context. We only add heavier search later if a channel's memory grows huge.

The thing we must get right Keeping channels separate. Channel A must never read Channel B's memory. Every memory access is strictly fenced to its own channel — get this wrong and private info leaks across teams.

8Use cases, step by step

How real situations actually play out. Click any to expand. The tag shows which build phase unlocks it.

💬 Ask it anything in any channel Phase 0›

In #payments, an engineer types: "@Claude what changed in the refund flow last week?"

  1. Slack delivers the message; the bot sees it's a direct tag → the Gate says "act."
  2. The Orchestrator loads #payments's identity (its instructions + allowed tools) and its memory snapshot.
  3. The agent reads the thread, calls the GitHub tool to list recent commits to the refund code, and reads the relevant files.
  4. It posts a plain-English summary back into the thread. Everyone in the channel can see it.
SlackGateAgentIntegrationsMemory
🐞 Triage a customer-support bug (the flagship) Phase 0›

A bug report lands in #support: "Customer X can't load their asset list." Someone tags @Claude. (This is what Henry already does today — we're just generalizing it.)

  1. The agent pulls the full ticket from the help-desk tool (error messages, screenshots, the customer's account ID).
  2. It searches the session-replay tool for that user's failing requests around the error time.
  3. It searches the GitHub codebase for the code path behind the failing request and reads it.
  4. It lines up the timeline, identifies the likely root cause, and posts a clear investigation summary in the thread — with file links.
AgentIntegrationsHelp-deskGitHubLogs
🤝 Hand off a half-finished task to a teammate Phase 1 · memory›

Alice asks @Claude to investigate a flaky test, then logs off. Two hours later Bob picks it up in the same channel.

  1. While working for Alice, the agent writes what it found into the channel's memory notebook (the cause it suspects, what it ruled out, what's left).
  2. Bob types: "@Claude where did you get to on that flaky test?"
  3. The Orchestrator loads the same channel memory — so the agent already knows the full story. Bob doesn't re-explain anything.
  4. The agent continues from where it left off and posts the next step. Because everyone shares one bot per channel, the handoff is seamless.
MemoryOrchestratorMultiplayerAgent
🔔 Jump in on its own (proactive / ambient) Phase 2 · ambient›

A customer question sits unanswered in #support for a day, then someone says "any update?" — but the original owner is out.

  1. The bot is reading every message (it already does), but normally stays silent.
  2. The Gate's small, cheap model flags this thread as "stalled, has an open question" — worth a nudge.
  3. To avoid being annoying, it checks a cooldown (it won't nudge the same thread repeatedly) and that ambient mode is switched on for this channel.
  4. It posts a gentle notify: "This thread's been waiting a day — here's a draft answer from our docs, want me to send it?" It suggests; a human approves.
GateAmbientCooldownMemory
⏳ Keep working over hours (or on a schedule) Phase 3 · scheduler›

"@Claude every morning at 9am, summarize overnight errors and post them here." Or: "keep monitoring this incident and update me as it changes."

  1. The agent writes a durable task into the scheduler's to-do list in the database (not just held in memory).
  2. A background worker wakes up on schedule — and this survives the app restarting, unlike a quick one-off job.
  3. Each run loads the channel's memory, does the work (e.g., query the error tool), and posts an update.
  4. For long jobs, the agent can even schedule its own follow-up steps and continue across hours or days.
SchedulerDurable tasksMemoryAgent
🔐 Connect a new tool, safely Phase 4 · governance›

An admin wants the #growth channel's bot to be able to read (but not write) the analytics warehouse, and nothing else.

  1. The admin enables the "analytics" integration only for #growth, read-only, in that channel's identity/config.
  2. First time the agent uses it, it triggers the private OAuth flow (DM a login link, pause, resume) tied to the requesting person.
  3. Every call is checked against the channel's permissions and recorded in the audit log: who asked, what ran, what it cost.
  4. If the channel hits its spending cap, the bot stops and tells the admin — no surprise bills.

10Build roadmap

Each phase is useful on its own. We ship in this order.

0
Works in any channel + any model

Move off the Claude-only toolkit to Pydantic AI + LiteLLM. Drop the single-channel limit. Put tools behind the Integration contract. ← our V1 starts here

1
Per-channel memory

Add the database, the memory notebook + snapshot, and the read/write-memory tools. Unlocks "it remembers us" and seamless handoffs. ← V1 ends here

▶
Sandboxed "run code" tool (core)

Give the bot a locked room so it can run code — test a fix, run a query — using the lightweight built-in sandbox, swappable to E2B / Daytona / Modal.

2
Proactive / ambient mode

The cheap Gate, notify-only behavior, cooldowns, and follow-ups on forgotten threads.

3

A real to-do list in the database + a background worker, so the bot can work over hours/days and survive restarts.

4
Governance & multiplayer polish

Per-channel/company spend caps, audit log, tool scoping, and the connect/disconnect admin surface.

11Plain-English glossary

Every bit of jargon used above, in one line each.

Agent
An AI that doesn't just chat — it plans, uses tools, reads the results, and repeats until a task is done.
Agent loop
That plan → act → observe → repeat cycle the agent runs.
Tool
An action the AI can take — "search GitHub," "read this ticket," "query the database."
Pydantic AI
The Python library that runs our agent loop. Works with any model.
LiteLLM
A translator that lets one piece of code talk to 100+ AI models the same way — our "any model" layer.
MCP
"Model Context Protocol" — an open standard for plugging external tools into AI agents. Vendor-neutral.
Slack Bolt
Slack's official toolkit for building bots. Handles incoming messages and posting replies.
The Gate
A cheap, fast filter that decides whether the bot should respond — especially for unprompted (ambient) replies.
Ambient / proactive
The bot acting on its own — flagging things, nudging stalled threads — without being tagged.
Per-channel memory
A separate, growing notebook of facts and tasks for each Slack channel.
Multiplayer
One shared bot per channel that everyone talks to — so anyone can pick up where someone else left off.
Durable task
A job saved in the database so it survives restarts and can run later or for a long time.
Sandbox ("locked room")
A sealed, throwaway computer where the bot can run code safely — it can't touch your real files or other teams' data.
Taking actions vs. running code
Pressing an app's official button via its API (a ticket, a PR) is "taking an action" — guarded by permissions. Writing and running a program is "running code" — guarded by the sandbox.
OAuth
The standard "Log in with…" approval flow. Lets the bot act as you on a tool without ever seeing your password.
Integration
A bundle of a tool + how it logs in + a hint on how to use it. Our pluggable unit.
Governance
The safety controls: spending limits, audit logs, and which channel may use which tools.
Self-hosted
You run it on your own computers, so your data never leaves your control.
