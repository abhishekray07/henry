Plug any tool into Henry with MCP

Instead of hand-coding a new Python integration every time Henry needs to reach a new system (HelpScout, Linear, Sentry…), you list servers in one config file and flip them on per channel.

Codex reviewed · 22 findings folded in Not yet implemented Source: docs/plans/2026-07-13-mcp-integrations.md

Why now

The HelpScout moment

You asked @Henry help me investigate: <a HelpScout link> and Henry answered, correctly, that he has no way to open it. Today Henry has exactly two tools baked in — GitHub and generic web fetch — and each one is hand-written Python living inside Henry's codebase. Every new system means writing, testing, and maintaining more of that code.

MCP (Model Context Protocol) is the industry's standard plug socket for AI tools. Thousands of ready-made MCP "servers" exist — small programs that expose a system's actions (read a ticket, search an issue tracker, query a database) in a format any AI agent can consume. Henry's original design doc already promised this ("External tools = MCP toolsets"); it just never got built. This plan builds it.

The shape of it

What happens when someone @mentions Henry

Nothing changes until the last step — that's the point. MCP slots into the existing per-channel gating instead of inventing a new mechanism.

Slack mention@Henry in a channel
→
Channel configeverything on by default; per-channel exceptions
→
Agent runfresh agent per request, as today
→
Builtin toolsgithub · web · memory · sandbox
+
MCP serversnew — from mcp.json, e.g. helpscout

An MCP server runs either as a small subprocess on your machine (command) or as a remote HTTPS endpoint (url). Henry connects, asks it what tools it offers, and hands those tools to the model — namespaced as helpscout_get_conversation etc. so servers can't collide.

Operator experience

One config file: mcp.json

Same format Claude Desktop uses, so most servers' READMEs paste straight in. Secrets stay in .env and are referenced with ${VAR}.

{
  "mcpServers": {
    "helpscout": {
      "url": "https://example.com/mcp",
      "headers": {"Authorization": "Bearer ${HELPSCOUT_API_KEY}"},  ← from .env
      "description": "Read Help Scout conversations",       ← goes into Henry's prompt
      "tools": ["get_conversation", "search_conversations"], ← allowlist: only these two
      "on_tool_error": "error"                               ← never auto-retry actions
    },
    "filesystem": {
      "command": "npx",                                      ← local subprocess server
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/data"]
    }
  }
}

And that's the whole setup. The OSS version is single-tenant — you are the operator, the admin, and the user — so adding a server here enables it in every channel automatically (a new "*" default). No admin UI, no per-channel SQL. A channel can still be scoped down with an explicit channel_config row when you want an exception, like keeping a write-capable server confined to an ops channel.

The decision the review changed

Connection lifecycle: the one hard problem

The first draft connected to every server at startup and held the connections open. Two independent reviews (an internal design pass, then Codex reading the actual library source) proved that design couldn't work. This is the plan's most important revision:

Rejected — draft 1

Connect at startup, hold forever

Open every configured server when Henry boots; keep connections pinned for the process lifetime.

Why it dies: the library counts how many things hold a connection, and only reconnects when that count hits zero. Pinning means it never hits zero — so a crashed server stays dead until you restart Henry. The "reconnect guard" patch had no place to run from and broke under concurrent runs.

Also: it spins up servers no channel ever enabled.

Adopted — final

Connect per run, keep processes warm

Each agent run opens the connections it needs and releases them when done — the library already does this on its own.

Why it works: the count returns to zero after every run, so a crashed server heals on the next message automatically. Local subprocess servers stay warm between runs (keep_alive), so there's no restart cost — just a quick handshake.

Cost: remote servers re-handshake each run. Fine for V1.

One consequence needs explicit handling: "keep the process warm" means shutdown must deliberately kill those subprocesses — closing the connection alone doesn't. Henry's shutdown path gets that, and a pre-existing bug gets fixed along the way (a failure during startup could already leak database and HTTP connections; now everything after runtime construction sits inside the cleanup scope).

Trust model

Security: five layers, honestly labeled

  • Definitions are operator-only, by design. A server definition means "run this command on my machine" or "send this key to this URL" — that power stays with whoever controls mcp.json and .env, never a chat command. In single-tenant that's you; servers are then on everywhere by default, with per-channel scoping as the exception.
  • Tool allowlist per server — the primary control. Henry can't know which of a third-party server's tools delete things — so "tools": [...] lets you expose only the read-only ones. The docs will say: always allowlist third-party servers.
  • No auto-retry of failed actions. Retrying a failed "send reply" can send it twice. Errors surface instead of retrying, unless a server is explicitly marked safe to retry.
  • Server instructions ignored. MCP servers can ship their own "system prompt" text — the single biggest injection channel. Henry hardcodes this off.
  • Output sanitized — with a scoped claim. Every string a server returns (including nested ones) gets Henry's framing-tag escaping and a ~50KB size cap.
Honest caveat: sanitization protects Henry's prompt structure, not against a malicious server writing persuasive English ("ignore your instructions and…"). The real defenses are the opt-in and the allowlist. The docs say exactly this rather than implying more.

Execution

Ten steps, test-first

Each step is a failing test, then the code that makes it pass. The order is deliberate: contracts first, riskiest proof last.

  1. The "*" enablement default — everything configured is on everywhere; an explicit channel row still narrows. The single-tenant decision, in code.

  2. Capability protocol — the ToolsetProvider contract. Foundation both later steps depend on.

  3. Runner accepts toolsets — proven with a plain local toolset, no MCP involved yet. Isolates the runner change from MCP entirely.

  4. Config parsing — every validation rule, env expansion, missing-file semantics.

  5. The adapter — server definition → integration, allowlists, timeouts, sanitizer, idempotent close.

  6. Settings + dependency declaration.

  7. Wiring — registry merge, shutdown ordering.

  8. App lifecycle scope — the leak fix, with a test that injects a startup failure.

  9. End-to-end against a real MCP server — a tiny checked-in server; tests cover the risky parts: kill the server process → next run heals; shutdown really terminates the subprocess; the allowlist really filters. The proof the whole plan hangs on.

  10. Docs — mcp.json.example + README, including the honest security caveat.

Done means: full test suite green, then a live smoke test — a real server in mcp.json, enabled for one test channel, exercised from Slack with an @mention.

Provenance

How this plan was hardened

Draft 1 was designed from two codebase explorations (Henry's internals + the installed pydantic-ai library), then challenged twice: an internal architecture review reshaped the integration contract, and a Codex review — which read the actual library source, ~1.5M tokens of it — found 18 issues including the fatal reconnect flaw, plus 4 gate findings (2 critical). All 22 are resolved in the current plan; the two review sections above (lifecycle and security) are where most of that pressure landed. A third revision followed a product decision: the OSS version is single-tenant, so enablement flipped from per-channel opt-in to on-everywhere by default. That revision post-dates the Codex review; engineering review hasn't run yet.
